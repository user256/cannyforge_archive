# Ticket 208: Refactor inline CSS to enqueued stylesheets

**Sprint:** 2 — Hardening & fit
**Status:** Not started
**Owner:** unassigned
**Estimate:** S

---

## Context

Currently, the inline CSS variables generated by `ThemeCssBuilder` are either appended directly to the DOM or added via style attributes. This approach, while functional for an MVP, violates strict Content Security Policies (CSP) and deviates from standard WordPress asset management practices.

## Goal

Refactor the theme's dynamic CSS delivery to use standard WordPress enqueue functions (`wp_add_inline_style`) attached to a registered stylesheet handle.

## Acceptance criteria

- [ ] The plugin registers and enqueues a primary CSS stylesheet for the frontend using `wp_enqueue_style`.
- [ ] `ThemeCssBuilder` outputs CSS that is then attached to this stylesheet handle via `wp_add_inline_style()`.
- [ ] No `style="..."` attributes are manually injected for the overall theme layout variables in the HTML output unless necessary for specific elements.
- [ ] Existing front-end theming controls and layout logic continue to work without visual regressions.

## Out of scope

- Redesigning the front-end components or the existing CSS design system.
- Minifying or compiling CSS (which is handled by other build steps or plugins).

## Dependencies

- **Blocks:** none
- **Blocked by:** none
- **External:** none

## Approach (optional)

Update `src/Frontend/ArchiveAssets.php` where CSS is registered. Ensure the `ThemeCssBuilder` returns a valid CSS string block, and pass it to `wp_add_inline_style('cannyforge-archive-frontend', $css)`.
